# Contract: Release Workflow Interface

**Feature**: 026-github-release-action  
**Contract Type**: GitHub Actions Workflow Interface  
**File**: `.github/workflows/release.yml`  
**Date**: 2026-03-31

---

## Overview

This contract defines the public interface of the `release.yml` GitHub Actions workflow — what triggers it, what inputs it accepts, what permissions it requires, and what outputs it produces.

---

## Triggers

### Automatic: `workflow_run`

| Property | Value |
|----------|-------|
| Triggering workflow name | `Android CI` |
| Triggering event type | `completed` |
| Branches filter | `main` only |
| Required conclusion | `success` (enforced via `if` condition on the release job) |

**Effect**: The release pipeline runs automatically every time `Android CI` completes successfully on the `main` branch.

### Manual: `workflow_dispatch`

Invokable by any user with write access to the repository from the repository's Actions tab.

| Input | Type | Default | Description |
|-------|------|---------|-------------|
| `prerelease` | `boolean` | `true` | Mark the resulting release as a pre-release. Defaults to `true` to protect against accidentally publishing a production release from a manual trigger. |

**Effect**: Builds and publishes a release from the current HEAD of the selected branch. If the branch is not `main` OR if `prerelease` is `true`, the release is marked as a GitHub pre-release.

---

## Required Permissions

The workflow declares these permissions at the job level:

| Permission | Scope | Reason |
|------------|-------|--------|
| `contents: write` | Repository contents | Create/push the version commit, create the git tag, and publish the GitHub Release with assets |

No additional secrets beyond the built-in `GITHUB_TOKEN` are required for the initial scope (debug-signed APK).

**Branch protection prerequisite**: The `github-actions[bot]` actor must be permitted to push to the `main` branch for the version commit-back step to succeed. If this cannot be permitted, the fallback `v{versionName}-{versionCode}` tag format is used and the commit-back step is skipped.

---

## Job Dependency Contract

| Dependency | Enforcement |
|------------|------------|
| `Android CI` `preflight` job success | `workflow_run` trigger only fires when the full `Android CI` workflow concludes with `success`. The release job additionally checks `github.event.workflow_run.conclusion == 'success'` in its `if` condition. |
| Own internal preflight | The release job runs `verify-android-prereqs.ps1` before the build step |
| Release hardening verification | `./gradlew verifyReleaseHardening` must exit 0 before `assembleRelease` runs |

---

## Build Contract

| Property | Value |
|----------|-------|
| Gradle task | `assembleRelease` |
| Build type | `release` |
| Minification | `isMinifyEnabled = true` (enforced by `verifyReleaseHardening`) |
| Resource shrinking | `isShrinkResources = true` (enforced by `verifyReleaseHardening`) |
| ProGuard rules | `proguard-android-optimize.txt` + `app/proguard-rules.pro` |
| Signing | Debug signing (initial scope) |
| APK output path | `app/build/outputs/apk/release/ndi-for-android-{versionName}.apk` |

---

## Output Contract

### Git Tag

| Property | Value |
|----------|-------|
| Format | `v{versionName}` where `versionName` comes from `version.properties` post-build |
| Example | `v0.12.4` |
| Uniqueness | Tag must not already exist; workflow fails if duplicate detected |

### GitHub Release Entry

| Property | Value |
|----------|-------|
| Release name | `NDI for Android {versionName}` |
| Release notes | Auto-generated by GitHub from merged PRs and commits since the previous release tag |
| Attached assets | Exactly one: `ndi-for-android-{versionName}.apk` |
| Pre-release flag | `false` for automatic `workflow_run` triggers; controlled by trigger and input for `workflow_dispatch` |

### Version Commit (commit-back)

| Property | Value |
|----------|-------|
| Commit message | `chore: bump version to {versionName} (code {versionCode}) [skip ci]` |
| Changed file | `version.properties` |
| Branch | `main` |
| Author | `github-actions[bot]` |

## Failure Modes and Expected Behavior

| Failure Scenario | Expected Behavior |
|-----------------|-------------------|
| `Android CI` failed on main | Release workflow does not start (workflow_run only fires on success) |
| Preflight (`verify-android-prereqs.ps1`) fails | Pipeline fails before build; no release created; error log identifies missing SDK/tool |
| `verifyReleaseHardening` fails | Pipeline fails before build; error message identifies which hardening flag was disabled |
| `assembleRelease` Gradle build fails | Pipeline fails; no release created; Gradle build log is available in Actions artifacts |
| APK file missing or empty after build | Pipeline fails at artifact verification step; log reports expected path and actual state |
| Duplicate git tag exists | `softprops/action-gh-release` fails with non-zero exit; release is NOT overwritten; error log names the duplicate tag |
| `GITHUB_TOKEN` lacks `contents: write` | Release creation fails at the publish step; log identifies permission error |
| Bot push to main blocked by branch protection | Version commit-back step logs a warning; release publication continues; follow-up manual version commit required |

---

## Invariants

1. No release is ever published if `assembleRelease` did not produce a non-empty APK at the expected path.
2. No release is ever published if any preflight or hardening gate failed.
3. A released APK always has R8/ProGuard minification and resource shrinking enabled.
4. Release tags are always prefixed with `v` and use the semver format from `version.properties`.
5. Releases from non-main branches are always marked as pre-releases.
